# @asochan-ai Skills — 詳細ガイド

## なぜこのスキルパックが必要なのか

### 問題: AIコーディングの「いきなり実装」問題

Claude Codeは優秀だが、指示が曖昧だと**いきなりコードを書き始める**。
これが起こす問題:

```
❌ よくある失敗パターン

1. 「ログイン機能作って」
   → Claude: DBスキーマ作成、API実装、フロント実装...
   → あなた: 「OAuth対応してほしかったんだけど...」
   → 全部やり直し（2時間の無駄）

2. 「決済機能追加して」
   → Claude: Stripe組み込み完了！
   → あなた: 「既にPayPal連携のコードあったんだけど...」
   → 重複実装（既存コードを見てなかった）

3. 「管理画面作って」
   → Claude: 完成！
   → セキュリティレビュー: 「認証なしで全データ見えますが...」
   → 致命的な手戻り
```

### 解決策: 「設計→承認→実装」の強制

```
✅ このスキルパックを入れると

1. 「ログイン機能作って」
   → super-plan: 「OAuth? メール/パス? SSO? 要件を確認させてください」
   → 設計書で合意してから実装
   → 手戻りゼロ

2. 「決済機能追加して」
   → super-plan Agent A: 「既にPayPal連携コードが payments/paypal.ts にあります」
   → 既存コードを拡張する設計
   → 重複ゼロ

3. 「管理画面作って」
   → super-plan: 設計書にセキュリティセクション含む
   → security-audit: 実装後に3段階チェック
   → 脆弱性ゼロ
```

## ベネフィット一覧

### 1. 手戻りが激減する

| 指標 | スキルなし | スキルあり |
|------|-----------|-----------|
| 設計変更による手戻り | 頻繁（曖昧な指示→間違った実装） | ほぼゼロ（設計書で合意済み） |
| 既存コードとの重複 | よくある（AIが既存コードを知らない） | ゼロ（Agent Aが全コードを調査） |
| セキュリティ手戻り | 後から発覚 | 設計段階で対策済み |

### 2. 既存コードを無駄にしない

super-planのAgent Aが**プロジェクト内の全コードをGlob+Grepで検索**する。
「この機能、もう半分実装されてますよ」と教えてくれる。

```
Agent A の出力例:

再利用コード一覧:
- src/auth/oauth.ts:15-80 — OAuth基盤（Google対応済み） → コピーして拡張
- src/utils/validation.ts:1-50 — 入力バリデーション → import
- src/db/users.ts:20-45 — ユーザーテーブル定義 → 参照
```

### 3. エビデンスベース（推測禁止）

設計書の**全ての主張にソース（ファイルパス or URL）が必須**。
「たぶんこうだと思います」は許されない。

```
❌ エビデンスなし:
  「React Queryを使うのが良いでしょう」

✅ エビデンスあり:
  「React Queryを使う — 根拠: package.jsonに既にインストール済み（v5.2.1）、
   src/hooks/useApi.ts で既に3箇所で使用中。統一するのが合理的」
```

### 4. セキュリティが設計段階で入る（Shift-Left）

```
従来:  コード完成 → 「セキュリティ大丈夫？」→ 穴が見つかる → 手戻り
今:    設計段階でセキュリティセクション作成 → 実装時に対策込み → 完成時に監査
```

### 5. 品質が「自動で」保たれる

品質ガードスキル（pre-check, quality-rules, auto-test等）は**バックグラウンドで自動実行**。
あなたが意識しなくても、Claudeが勝手にチェックしてくれる。

## 仕組み（技術的な詳細）

### super-plan の6ステップ

```
Step 1: スコープ定義
├── 1a. 何を作る？完了条件は？
├── 1b. 誰が使う？品質で何を重視？
└── 1c. Non-Goals のストレステスト
        「本当にスコープ外でいい？」を敵対的に検証

Step 2: 徹底リサーチ（4エージェント並列）
├── 2a. 内部リサーチ（先に実行 — 確証バイアス防止）
│   ├── Agent A: 既存コード調査（Glob + Grep）
│   └── Agent B: 過去の教訓・決定事項の調査
│
└── 2b. 外部リサーチ（2aの結果を入力として実行）
    ├── Agent C: 技術ベストプラクティス（WebSearch）
    └── Agent D: 受け手/UX最適化リサーチ

Step 3: 設計書生成（テンプレートに従い構造化）
├── 概要 / ゴール / Non-Goals
├── リサーチサマリー（エビデンス付き）
├── アーキテクチャ / モジュール分割
├── タスク分解（YAML形式、依存関係付き）
├── セキュリティ & プライバシー
└── リスク & 緩和策

Step 4: エビデンスチェック（BLOCKING）
  全主張にソースがあるか機械的に検証。
  欠けていたら STOP → ギャップを埋める

Step 5: 教材生成（オプション）
  設計書の技術判断を理解しやすい形式に変換

Step 6: 承認ゲート
  あなたに設計書を提示。「OK」をもらうまで実装に進まない
```

### rapid-build の仕組み

super-planの設計書を入力として、以下の原則で実装:

```
原則1: 再利用 > 再発明
  Agent Aが見つけた既存コードをベースにする。ゼロから書かない

原則2: Config-First
  色、URL、キー、テキスト → 全て設定ファイルに外出し
  ハードコードゼロ

原則3: 1モジュール = 1責務
  変更理由が1つだけのモジュール分割（SOLID原則）

原則4: 品質ゲート
  各モジュール完成時にチェック → パスしたら次へ
```

### 品質ガードの多層防御

```
時系列で見ると:

作業開始
  │
  ├── pre-check (L1)
  │   「影響範囲: 5ファイル、適用ルール: 3件、過去インシデント: 1件で開始します」
  │
  ├── quality-rules (L2) ← 実装中ずっと並走
  │   コードを書く時点でルール違反を防ぐ（書いてから直すのではなく）
  │
  ├── auto-test
  │   テストフレームワークを自動検出して実行
  │
  ├── security-audit
  │   3段階: パターンスキャン → 依存関係 → AI攻撃者分析
  │
  └── verify
      構造・タスク・セキュリティ・gitを一括検証

完了
```

## 具体的な活用シナリオ

### シナリオ1: Webアプリの新機能追加

```
あなた: 「ユーザープロフィール編集機能を追加したい。super-planで設計して」

super-plan:
  → 「完了条件は？」「アバター画像は？」「メール変更は再認証必要？」
  → Agent A: 既存のUserモデル、APIルート、フロントコンポーネントを全調査
  → Agent C: プロフィール編集のUXベストプラクティスを調査
  → 設計書生成（モジュール分割、API設計、DB変更、テスト計画付き）
  → あなたが確認して「OK」

rapid-build:
  → 既存コードを拡張する形で実装
  → quality-rules が並走（バリデーション漏れを防止）
  → auto-test でテスト実行
  → security-audit で入力サニタイズ確認
```

### シナリオ2: 既存プロジェクトのセキュリティ強化

```
あなた: 「security-auditして」

security-audit:
  → Step 1: プロジェクト自動検出
  → Step 2a: シークレットパターンスキャン（APIキー露出等）
  → Step 2b: 依存関係チェック（npm audit / pip audit）
  → Step 3: 3つのAI攻撃者が並列でコードを攻撃
     - Agent A: フロントエンド攻撃（XSS, CSRF等）
     - Agent B: バックエンド攻撃（SQLi, 認証バイパス等）
     - Agent C: 信頼境界攻撃（SSRF, 権限昇格等）
  → レポート生成
```

### シナリオ3: プレゼン資料の自動生成

```
あなた: 「来週の会議用に新機能の提案スライド作って」

presentation-architect:
  → ブランド設定を読み込み
  → 構成設計（1スライド = 1メッセージ）
  → frontend-slides でHTML生成（12スタイルプリセット）
  → 品質ゲート（フォント最小18px、赤テキスト禁止等）
```

## 設定のカスタマイズ

各スキルはConfigセクションを持っており、プロジェクトに合わせてカスタマイズ可能:

```yaml
# quality-rules の例
rules_file: ".claude/quality-rules.yaml"   # 品質ルール定義ファイル
max_rules: 50                               # ルール上限
archive_threshold: "6 months"               # 違反ゼロでアーカイブ

# auto-test の例
test_timeout: 120                           # テストタイムアウト（秒）
coverage_flag: "--coverage"                 # カバレッジ取得フラグ
```

## 注意事項

- super-planは**4ファイル以上に影響する変更**で最も効果を発揮する。1ファイルの修正には過剰
- 品質ガードスキルは入れておくだけで自動で動く。明示的に呼ぶ必要なし
- スキルはMarkdownファイルなので、中身を読んでカスタマイズ可能
- SKILL.mdはオープンスタンダード。Claude Code以外のツールでも動作する
