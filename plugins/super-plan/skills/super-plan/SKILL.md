# Super Plan スキル

コードを書く前に、エビデンス付きの包括的な設計書を作成する。
開発時間を浪費する最大の原因「リサーチ不足のまま実装を開始すること」を防止する。

## 設計原則

| 原則 | 出典 | 適用 |
|------|------|------|
| 設計書ファースト | Google Engineering | コードの前に設計書を書く。紙の上の判断は安い。コードの変更は高い |
| 再利用 > 再発明 | Unix哲学 | 既に動いている既存コードを見つける。コピーして修正する。ゼロから書かない |
| エビデンスベースの計画 | 実証主義 | 設計書の全主張にソース（ファイルパス、URL、行番号）が必要。推測禁止 |
| 単一承認ゲート | — | 不完全な複数イテレーションではなく、1回の徹底レビュー |
| Shift-Left Security | OWASP | 実装後ではなく設計段階で攻撃面を特定する |

- 新機能、モック、デモ、プロトタイプ
- クライアント向け成果物
- アーキテクチャ判断が必要なタスク

```yaml
# 品質閾値

```

## ワークフロー

### Step 1: スコープ定義 + ペルソナ + ストレステスト

#### 1a. 基本スコープ（AskUserQuestion）

以下を明確にする:
- 何を作るか？（1-2文の説明）
- 「完了」とは何か？（測定可能な成功基準）
- 何をスコープ外にするか？（Non-Goals）

プロジェクトにアダプタファイル（例: `super-plan-adapter.md`）がある場合、
ここで読み込んでプロジェクト固有の設定を適用する。

**失敗時: STOP → 成功基準が曖昧ならAskUserQuestionで再質問。推測で進めない**

#### 1b. 受け手ペルソナ + 品質属性（AskUserQuestion）

成果物の「受け手」を明確にする。技術的に正しくても、受け手に合わなければ無価値。

- **誰が受け取るか？**（ペルソナ）
  - 技術レベル: エンジニア / 非エンジニア / 混在
  - 前提知識: 何を知っていて、何を知らないか
  - 利用シーン: いつ・どこで・どう使うか
- **品質属性は？**（優先順位付きで選択）
  - 正確さ / 読みやすさ / エンゲージメント / 速度 / 保守性 / セキュリティ

> 背景: 学習システム設計（2026-03-17）で「小学生でもわかるレベル」「読み方ルビ必須」等の
> 品質要件がStep 1で聞けておらず、設計書完成後にユーザーフィードバックで3回手戻りが発生。

#### 1c. スコープストレステスト

Non-Goalsを定義した後、各Non-Goalに対して「本当にスコープ外でよいか？」を検証する。

**手順**: 各Non-Goalに対し、以下の敵対的シナリオを当てる:
1. **ステークホルダーシナリオ**: 投資家DD / 監査 / クライアント説明で問われたら？
2. **障害シナリオ**: このNon-Goalが原因で事故が起きたら、説明責任は取れるか？
3. **スケールシナリオ**: 6ヶ月後に必要になった場合、今の設計で追加可能か？

**判定**: 1つでも「説明できない」シナリオがあれば、Non-GoalをGoalに格上げするか、
段階的対応（Phase 2）として明示的にロードマップに入れる。

**失敗時: STOP → ストレステストで不安が残るNon-GoalはAskUserQuestionでユーザーに確認**

> 背景: 学習システム設計で「フルスタックエンジニアの知識全体をカバーしない」をNon-Goalにしたところ、
> ユーザーから「これで投資家に説明責任負えるか？」と指摘。4層モデルへの拡張が必要になった。

### Step 2: 徹底リサーチ（2段階: 内部先行 → 外部補完）

#### Step 2a: 内部リサーチ先行（Agent A + B を並列起動）

**目的**: 設計書の「前提」を確定する。外部リサーチの前に、上流ドキュメントで既に定義されている事業モデル・戦略・制約を正確に把握する。

**BLOCKING**: Step 2aの結果を「前提サマリー」として文書化してからStep 2bに進む。
前提サマリーには以下を含める:
- 事業モデル（何を誰に売るか）
- 収益モデル（どう稼ぐか）
- チーム構成
- 既に決定済みの戦略・方針
- ユーザーが「定義済み」と言った項目の該当箇所（パス:行番号）

**Agent A: 既存コード調査**
- Glob + Grepで関連する全ソースファイルを検索
- 各ファイルを完全に読む（ヘッダーだけでなく全体）
- 出力:
  1. 「再利用コード一覧」— 問題の一部を既に解決している関数/モジュール
     形式: `{パス}:{行範囲} — {何をするか} — {再利用方法: コピー/import/参照}`
  2. 「ベースファイル一覧」— 出発点としてコピーすべきファイル（再利用 > 再発明）
     形式: `{元ファイル} → {先ファイル} — {このプロジェクトで変更する点}`

**Agent B: 内部ナレッジ調査**
- 過去の意思決定、インシデント報告、教訓をプロジェクト内で検索
- ドメインに関する既存ドキュメントを検索
- **Deployment Target調査**（必須）:
  1. 「どのリポに作るべきか？」— 拡販/プロダクト=共有リポ、個人運用=個人リポ。worktree-workflow.mdの自動判定テーブルを参照
  2. 「外部通知があるか？」— Slack/Email/Webhook → テスト環境（テストチャネル等）の特定
  3. 「必要なSecret/権限は？」— 既存のGitHub Secrets・Slack App設定を検索
- 出力:
  1. 「制約一覧」— 守らなければならない決定/ルール
     形式: `{制約} — 根拠: {ドキュメントパス}:{行番号}`
  2. 「教訓一覧」— 過去に避けるべきミス
     形式: `{教訓} — 根拠: {ドキュメントパス}`
  3. 「Deployment Target」— 実装リポ・テスト環境・通知先・必要Secret
     形式: `{カテゴリ}: {値} — 根拠: {判定ロジックまたはドキュメント}`

#### Step 2b: 外部リサーチ（Agent C + D を並列起動、Step 2aの前提サマリーを入力）

**目的**: 内部で確定した前提を「検証・補完・強化」する。他社の最適解や業界ベストプラクティスを参照し、上流の判断が最適か確認する。外部リサーチは内部の「代替」ではなく「補完」。

**Agent C: 外部リサーチ**
- WebSearchで技術的ベストプラクティス、リファレンス実装、ドメイン知識を調査
- **入力**: Step 2aの前提サマリーを受け取り、前提と矛盾する結果が出た場合は「矛盾フラグ」を出力に含める（設計書で解決する）
- 出力:
  1. 「技術判断」— 技術/アプローチの選択と根拠
     形式: `{判断} — 出典: {URL}`
  2. 「矛盾フラグ」（該当時のみ）— 外部知見と内部前提の不一致
     形式: `{内部前提} vs {外部知見} — 出典: {URL} — 要ユーザー判断`

**Agent D: 受け手・UXリサーチ**（Step 1bのペルソナ情報を入力とする）
- Step 1bで定義したペルソナに最適な情報構造・体験設計・品質基準をWebSearchで調査
- 調査内容:
  - ペルソナの技術レベルに合った説明手法（例: 非エンジニア向け → 比喩・図解・段階的説明）
  - 品質属性の業界ベストプラクティス（例: 正確さ → CER/Admiralty、エンゲージメント → Duolingo/Fireship）
  - 類似プロダクト/教材の成功パターンと失敗パターン
- 出力:
  1. 「受け手最適化」— ペルソナに合わせた設計指針
     形式: `{指針} — 根拠: {URL} — 適用先: {モジュール名}`
  2. 「品質基準」— 各品質属性の具体的な達成基準
     形式: `{属性}: {基準} — 出典: {URL}`

> 背景: 学習システム設計で「Fireship 100秒パターン」「Brilliant Try-Before-Read」
> 「Duolingoストリーク」等の受け手向け設計知見が初回リサーチに含まれず、
> ユーザーフィードバック後の追加リサーチで発見。最初から調査すべきだった。

**失敗時: STOP → いずれかのAgentが有用な結果を返さない場合、検索クエリを変えて再実行**

> 背景: Agent Native設計書（2026-03-18）でStep 2を全Agent並列実行した結果、
> 外部リサーチ（免許要件・成約報酬モデル）のフレームに引きずられ、
> 内部ドキュメント（README・STRATEGY）に明記されていたビジネスモデルを無視。
> 「Agent Native = 人材紹介会社」と誤定義し、ユーザーに指摘されて全面改訂。
> 原因: 内部の前提を確定する前に外部リサーチの結果を前提として採用した（確証バイアス）。

### Step 3: 設計書作成

PLAN_TEMPLATE.mdに従い、構造化された設計書を作成:

1. **概要** — 何を解決するか？
2. **ゴール / Non-Goals** — 明示的なスコープ境界（ストレステスト済み）
3. **受け手ペルソナ & 品質属性** — 誰のため？何を重視？（Step 1bの結果）
4. **リサーチサマリー** — Step 2の統合出力（エビデンス付き、Agent D含む）
5. **設計**
   - アーキテクチャ: 単一責任の原則に従うモジュール分割
     （各モジュールの変更理由は1つだけ）
   - 再利用計画: ベースラインとしてコピーする既存ファイル
   - 設定戦略: 環境依存の全値を外出し
     （12-Factor App: 色、URL、キー、テキスト、サイズ → 設定ファイル or 環境変数）
6. **タスク分解** — 機械読み取り可能なYAML形式（PLAN_TEMPLATE.md参照）
   - 各タスク: id, name, files, depends_on, done_criteria, base_file
   - 設定/セットアップタスクは依存関係なし（最初に実行）
   - 実装タスクは設定タスクに依存（順序を強制）
7. **セキュリティ & プライバシー** — モジュールごとの攻撃面と必要な対策
8. **テスト戦略** — 正しさの検証方法
9. **リスク & 緩和策** — 何が失敗しうるか

### Step 4: エビデンスチェックリスト（BLOCKING）

次に進む前に、設計書の全主張にエビデンスがあることを検証:

- [ ] 全ての「再利用」エントリにファイルパスと行範囲がある
- [ ] 全ての制約にソースドキュメントがある
- [ ] 全てのモジュールに1つの責務が明記されている
- [ ] 全てのタスクにテスト可能なdone_criteriaがある
- [ ] 設定戦略がハードコード可能な全値をカバーしている
- [ ] セキュリティセクションが外部入力を扱う全モジュールをカバーしている
- [ ] ベースファイル（再利用 > 再発明用）の存在が `test -f` で確認済み
- [ ] 受け手ペルソナが定義され、品質属性に優先順位がついている
- [ ] Non-Goalsがストレステスト済み（敵対シナリオ3つで検証）
- [ ] Agent Dの受け手最適化指針が設計に反映されている
- [ ] 設計書の事業モデル/前提が上流ドキュメント（STRATEGY_2027.md, README.md等）と矛盾していない
- [ ] ユーザーが「定義済み」と言った項目は、該当ドキュメントをReadして設計に正確に反映した

**エビデンスが欠けている項目があれば、STOP。続行前にギャップを埋める。**

通常の計画との決定的な違い: 曖昧な記述は許可されない。

### Step 5: 教材自動生成（plan-to-study）

設計書のエビデンスチェック通過後、**承認を求める前に**教材を自動生成する。
ユーザーが技術判断を理解してから承認判断できるようにする。

**手順:**
1. `plan-to-study` スキルを呼び出す（入力: 完成した設計書）
2. plan-to-studyが STUDY_GUIDE + QUIZ + NOTEBOOKLM の3形式を生成
3. 生成完了後、Step 6（承認ゲート）に進む

**このステップをスキップしない。** 設計書の技術判断を理解せずに承認すると、
実装中のフィードバック手戻りが発生する。

> 背景: Meeting Agentバックエンド設計書（2026-03-18）でplan-to-studyの提案を怠り、
> ユーザーに指摘された。根本原因: super-planのワークフローにplan-to-studyが組み込まれていなかった。

### Step 6: ステークホルダー承認ゲート

ステークホルダーに提示:
1. リサーチサマリー（何が見つかったか）
2. アーキテクチャ概要（どう構築するか）
3. タスク分解と見積もり
4. セキュリティプレビュー
5. リスク
6. 教材（Step 5で生成済み — ユーザーが先に学習できる）

**明示的な承認が得られるまで実装に進まない。**

## エスカレーション

自動化で完了できない場合:
1. どのステップで詰まったか・不足しているエビデンスが何かをユーザーに伝える
2. ユーザーが判断を下す（スコープ変更、技術選定、Non-Goal格上げ等）
3. ユーザー判断後、該当ステップから再開する

## セキュリティ

- **APIキー/トークン**: 不要（設計書の作成のみ）
- **ログ出力禁止**: 該当なし
- **PII取り扱い**: 設計書にPIIを含めない。クライアント名等は匿名化を検討
- **外部アクセス**: Agent CとDがWebSearchを実行。ドメイン制限なし

## 出力

PLAN_TEMPLATE.md形式に従った設計書。プロジェクト内に保存する。
この設計書が `rapid-build` スキルへの入力となる。

## 他のスキルとの連携

| スキル | 関係 | 説明 |
|--------|------|------|
| plan-to-study | Step 5で呼び出し | 承認前に教材を自動生成。ユーザーが技術判断を理解してから承認する |
| execution-strategy | 後工程 | super-plan完了後に自動で実行推奨される。最適な実行方法（cmux/Agent Team/Subagent）を判定 |
| rapid-build | 出力先 | 承認済み設計書を消費して実装する |
| security-scan | 参照 | Step 7（セキュリティセクション）で脅威モデリングのアプローチを参照 |
| parallel-tasks | 並列 | rapid-buildがタスク構造に基づきマルチエージェント呼び出しを判断 |
| skill-upgrade | 品質管理 | Q1-Q8基準でこのスキル自体の品質を検証 |
| quality-rules | 参照 | Step 2aでAgent Bがquality-rules.yamlを内部ナレッジとして読み込み。Step 4のチェックリストにquality-rulesの項目を自動追加 |
| leak-learner | 還流 | L5で蓄積されたルール・lessonsが次回のsuper-plan実行時に自動反映 |

## このスキルがやらないこと

- コードを書くこと（それはrapid-buildの仕事）
- エビデンスなしでアーキテクチャ判断をすること
- 「シンプルに見える」からリサーチをスキップすること
- ステークホルダー承認なしに進むこと
- 設計書のフォーマット決定（それはPLAN_TEMPLATE.mdの責務）

## 汎用性（Portability）

このスキルは **汎用フレームワーク** — エビデンスベースの設計書作成プロセス（スコープ定義・ペルソナ・内部/外部リサーチ・CERチェック・承認ゲート）はプロジェクト非依存。
他社は以下を変更するだけで使える:
1. Config の `plan_output_dir` を自社パスに変更
2. `PLAN_TEMPLATE.md` を自社のテンプレートに差し替え
3. アダプタファイル（`super-plan-adapter.md`）で検索パスをカスタマイズ

設計原則（Google Design Doc, SOLID, 12-Factor, Shift-Left Security）は業界標準。企業名ハードコードなし。

> 連携テーブルに leak-learner 行あり（Q10 PASS）。
