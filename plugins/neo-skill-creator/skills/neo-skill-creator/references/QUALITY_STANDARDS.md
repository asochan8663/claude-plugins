## スキル作成・品質標準（公式skill-creator + Zero Leak統合）

新規スキル作成・既存スキル改修時に適用する。公式Claude Code skill-creatorのベストプラクティスとZero Leak品質基準を統合。

### 作成フロー

「スキル作って」と言われたら:
1. **公式 skill-creator スキルを呼び出す**（Skill tool経由）— interview → scaffold のワークフロー
2. **本ルール（skill-standards.md）を公式フローに上乗せ適用** — Q1-Q10 + Zero Leak + Progressive Disclosure
3. 結果: 公式の構造品質 + うちの品質基準 = 最初からskill-upgrade合格品質

公式skill-creatorが scaffold したSKILL.mdに対して、本ルールのQ1-Q10を確認し、不足があれば追記する。

### ディレクトリ構造

```
skills/{name}/
├── SKILL.md          ← 必須。スキルの定義
├── lessons/           ← 必須。leak-learner (L5) の書き込み先
│   ├── *.md          ← Skill固有の教訓（leak-learnerが自動追記）
│   └── {client}.md   ← Client固有の教訓（あれば）
├── scripts/           ← 任意。確定的処理のスクリプト
├── references/        ← 任意。参照ドキュメント（必要時のみ読み込み）
└── assets/            ← 任意。テンプレート・アイコン等
```

**禁止**: README.md, CHANGELOG.md, LICENSE.md等の補助ファイルをスキル内に置かない

### SKILL.md のフォーマット

```yaml
---
name: skill-name
description: "1-2文。何をするか + いつ使うか。少し積極的に書く（トリガー漏れ防止）"
---
```

**frontmatterにはnameとdescriptionのみ。** 他のフィールドは入れない。

### Progressive Disclosure（3層情報設計）

| 層 | 内容 | サイズ目安 | 読み込みタイミング |
|---|---|---|---|
| L1: メタデータ | name + description | ~100語 | 常時コンテキストに存在 |
| L2: SKILL.md本文 | 手順・ルール・連携 | **500行以下** | スキルトリガー時 |
| L3: 参照リソース | scripts/, references/ | 制限なし | 必要時のみRead |

**SKILL.mdが500行を超えたら分割を検討。** references/に移せる情報はL3に降格。

### Q1-Q10 品質基準（skill-upgradeと同一基準 — 作成時に守れば事後チェック不要）

> **SSoT**: Q1-Q10の正式定義は `skill-upgrade/SKILL.md` を参照。以下はスキル作成時の簡易チェックリスト版。定義に差異がある場合はskill-upgrade/SKILL.mdが正。

スキル作成時に以下を満たすこと。skill-upgradeは事後検証だが、**作成時にここを守れば最初から合格品質になる。**

#### Q1: 設計原則テーブル
```markdown
| 原則 | 出典 | 適用 |
|------|------|------|
| {原則名} | {出典} | {このスキルでの適用方法} |
```
なぜ: 判断の根拠が明示されないと、改修時に意図が失われる

#### Q2: Config セクション
```markdown
## Config
| カテゴリ | キー | デフォルト値 | 説明 |
```
なぜ: 設定値がSKILL.md本文にハードコードされると、環境差異に対応できない

#### Q3: セキュリティ（API/認証/PII を扱うスキルのみ）
記載すべき内容:
- 使用するAPI/トークンの種類
- PII取り扱いルール
- ログ出力禁止項目

#### Q4: BLOCKINGゲート
```markdown
| Step | 失敗条件 | 動作 |
|------|---------|------|
| StepN | {条件} | STOP + {理由とリカバリ方法} |
```
なぜ: 失敗時に黙って続行すると、後工程で問題が拡大する

#### Q5: フェーズ分割
手順をStep 1, Step 2... に分割する。1つのStepに複数の責務を詰め込まない
なぜ: 粒度が粗いと、どこで失敗したか特定できない

#### Q6: エスカレーション手順
```markdown
| 状況 | 対応 |
|------|------|
| {自動化できない判断} | ownerに確認: 「{質問}」 |
```
なぜ: 自動化の限界を明示しないと、AIが勝手に判断して事故になる（V3: P9 勝手な変更 8件）

#### Q7: 他のスキルとの連携テーブル（**必須**）
```markdown
| スキル | 関係 | 説明 |
|--------|------|------|
| leak-learner | 学習 | owner指摘をlessons/に記録。2+スキル共通パターンはGlobal rulesに昇格 |
| {他のスキル} | {関係} | {説明} |
```
なぜ: スキルが孤立すると、同じ機能の重複実装が発生する。leak-learner行は全スキル必須

#### Q8: やらないこと
スコープ境界を明示。「このスキルがやること」の裏側を書く
なぜ: 書かないとスコープクリープが起きる（V3: P12 owner意図の誤解釈 5件）

#### Q9: 汎用性（該当するスキルのみ）
プロジェクト固有の値（パス、URL、名前）がSKILL.md本文にハードコードされていないか
なぜ: 別プロジェクトに持ち出せないスキルは再利用価値がない

#### Q10: Zero Leak L5 接続
- `lessons/` ディレクトリが存在する
- 連携テーブルに leak-learner 行がある
- スキル内に独自の学習ロジックがない（leak-learnerに委譲）
なぜ: 各スキルに学習エンジンを持たせると保守コスト30倍、ドリフトが発生する

### Zero Leak L5 接続（全スキル必須）

**全スキルに以下を含めること:**

1. `lessons/` ディレクトリの存在
2. 連携テーブルに leak-learner 行:
   ```
   | leak-learner | 学習 | owner指摘をlessons/に記録。2+スキル共通パターンはGlobal rulesに昇格 |
   ```
3. **スキル内に学習ロジックを書かない。** 学習エンジンはleak-learner 1つに集約

### 書き方のルール

**命令形で書く（公式推奨）:**
```
❌ 「ユーザーが〜した場合、〜することが推奨されます」
✅ 「〜の場合、〜する」
```

**WHYを書く（ALWAYS/NEVERより理由）:**
```
❌ 「絶対にハードコードしない」
✅ 「ハードコードするとownerが毎回確認する必要が生じる（V3: 49件）。config/envに外出しする」
```

**簡潔に（コンテキストは公共財）:**
- 効果のない記述は削除する
- 同じことを2回書かない
- 例は1-2個で十分（5個も要らない）

### description の書き方

descriptionはスキルリストに常時表示される。**トリガー漏れを防ぐため少し積極的に書く。**

```
❌ "YouTube動画の台本を作成します"
✅ "YouTube動画の台本を作成するスキル。「台本書いて」「スクリプト作成」等のトリガー、
   またはテーマ指定で起動。chara-tone連携で口調統一。"
```

### やってはいけないこと

- SKILL.mdに学習ロジックを書く（leak-learnerの責務）
- frontmatterにname/description以外を追加する
- 補助ファイル（README等）をスキル内に置く
- 500行超のSKILL.mdを放置する
- 連携テーブルなしのスキルを作る
- quality-rules.yamlのルールをSKILL.md内にコピーする（参照するだけ）

### 品質チェック

新規スキル作成後、`skill-upgrade` でQ1-Q10チェックを実行する。
全項目PASS or N/Aで合格。
